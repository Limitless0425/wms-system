## 四、核心业务流程

### 4.1 登录流程
1. 用户输入用户名密码 → `AuthController.login()`
2. BCrypt 校验密码 → `JwtUtil.generateToken()` 生成 JWT
3. 前端存入 `localStorage.token`，路由守卫检查后进入主界面
4. 每次请求由 `JwtAuthFilter` 拦截验证 Token

### 4.2 入库单创建流程
1. 选择供应商 → 后端返回该供应商所有零件
2. 零件列表以表格展示，每行独立选择：仓库、零件、数量、器具、库位
3. 批量导入支持文本解析（格式：`零件编码,数量,仓库编码,器具编码,库位编码`）
4. 库位容量校验（红色"超容量!"警告）
5. 提交后**自动生成看板**，状态直接变为 CONFIRMED

### 4.3 入库单修改流程
1. 点击修改 → 显示该入库单现有明细（非全量零件表）
2. 每行可独立修改：零件（下拉更换）、数量、仓库、器具、库位
3. 可删除行、添加新行
4. 保存后**自动清除旧看板并重新生成**所有未入库的新看板
5. 未改动的已入库看板不受影响

### 4.4 看板生成与扫码入库
1. 入库单创建/修改时**自动生成看板**，无需手动点击"生成"
2. 看板状态：PRINTED（待扫描）→ SCANNED（已入库）→ VOIDED（已作废）
3. **部分入库后自动生成剩余数量新看板**：扫完一张 → 剩余数量自动出新看板
4. 扫码入库写入 `InventoryRecord`，更新 `actualQty`，同步订单状态

### 4.5 入库单状态流转
```
DRAFT（草稿）→ 自动生成看板 → CONFIRMED（已确认）
→ 部分扫码入库 → PARTIAL（部分入库）
→ 全部扫码入库 → COMPLETED（已完成）
→ 作废 → VOIDED（已作废）
```

### 4.6 入库单删除
- 允许删除 DRAFT / CONFIRMED / PARTIAL / COMPLETED 状态
- 级联删除：库存记录 → 看板 → 入库单，完全清除痕迹

### 4.7 库存追溯
- 按零件追溯：输入零件编码 → 查询所有入库记录
- 按看板追溯：输入看板号 → 查询该看板对应的入库流水

## 八、Bug 修复记录

### 8.1 零件表格不显示
- **原因**：`el-table-column` 作用域插槽用 `#default="{r}"`，Element Plus 暴露的是 `{ row }`
- **修复**：全部改为 `#default="{ row }"`

### 8.2 入库单列表不显示
- **原因**：操作列 `#default="{ row:r }"` 重命名后 `v-if` 仍写 `row.status`
- **修复**：改为 `r.status`

### 8.3 仓库不支持编辑删除
- **修复**：后端新增 PUT/DELETE 接口，前端新增对应按钮

### 8.4 入库单乱码
- **现象**：部分持久化数据中文字段为 `?`
- **修复**：启动时根据主数据自动修复损坏字段

### 8.5 看板未入库状态冗余
- `NOT_INBOUND`、`SEALED`、`TRANSFER_INBOUND` 从未被后端设置
- **修复**：前端下拉只保留"待扫描/已入库/已作废"

|------|------|
